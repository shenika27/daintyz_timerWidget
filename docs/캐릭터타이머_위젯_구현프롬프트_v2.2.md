# 캐릭터 타이머 위젯 앱 - 구현 프롬프트 (v2.2)

## 0. 프로젝트 개요
안드로이드 홈 화면 위젯 형태의 "캐릭터 타이머" 앱을 만든다. 타이머가 진행되는 동안 위젯 안의 캐릭터가 런닝머신 위에서 뛰는 애니메이션을 보여준다. 기본 캐릭터는 무료, 추가 캐릭터는 스킨 형태로 유료 판매할 계획이므로 **스킨 시스템의 확장성을 최우선으로 설계**한다.

> 용어: 이 문서에서 **"1차 버전"** = 실제 출시(무료)까지 고려한 첫 번째 동작 버전을 의미. 결제 등 일부 기능은 1차 이후 단계에서 추가한다.

---

## 1. 기술 스택 (확정)

- **언어**: Kotlin
- **위젯 방식**: 전통적 `AppWidgetProvider` + `RemoteViews` (Jetpack Glance 미사용 — 1fps 프레임 교체 방식에 RemoteViews가 더 적합)
- **minSdk**: 26 (Android 8.0, 포그라운드 서비스 기준)
- **targetSdk**: 35
- **상태 저장**: `SharedPreferences` (타이머 1개 단순 구조이므로 Room 미사용)

---

## 2. 위젯 레이아웃 (확정)

- **위젯 전체 크기**: 2×3 셀 (가로 2 × 세로 3), 고정 크기 (리사이즈 대응은 추후 확장)
- **타이머 영역**: 2×1 (타이머 표시 + 조작 버튼)
- **캐릭터 영역**: 2×2 (캐릭터 + 런닝머신 배경)

### 2-1. 타이머 위/아래 배치 전환 (설정으로 선택)
타이머 영역과 캐릭터 영역의 상하 위치를 설정에서 전환할 수 있다. 기본값은 "타이머 위".

```
[layoutMode = top (기본)]      [layoutMode = bottom]
┌─────────────┐               ┌─────────────┐
│  타이머 + 버튼  │ ← 2×1          │             │
├─────────────┤               │   캐릭터/배경   │ ← 2×2
│             │               │             │
│   캐릭터/배경   │ ← 2×2          ├─────────────┤
│             │               │  타이머 + 버튼  │ ← 2×1
└─────────────┘               └─────────────┘
```

**구현 방식**:
- 레이아웃 XML을 두 벌로 준비: `widget_timer_top.xml`(타이머 위) / `widget_timer_bottom.xml`(타이머 아래). RemoteViews는 자식 동적 재배열이 까다로우므로 레이아웃을 분리해 선택 적용한다.
- 위젯 갱신 시 `SharedPreferences`의 `layoutMode`(`"top"` | `"bottom"`) 값을 읽어 해당 레이아웃으로 `RemoteViews`를 생성한다.
- 설정 변경 시 **즉시 위젯을 강제 갱신**한다(`AppWidgetManager.updateAppWidget` 호출 또는 갱신 브로드캐스트 발송). 갱신을 트리거하지 않으면 다음 자연 갱신 때까지 이전 배치가 남는다.
- 두 레이아웃 모두 동일한 버튼 id·캐릭터 ImageView id를 사용해, 상태/프레임 갱신 로직이 배치와 무관하게 동작하도록 한다.

## 3. 핵심 기능 요구사항

### 3-1. 위젯 상태 (4상태 — 확정)

| 상태 | 상단(2×1) 화면 구성 | 하단(2×2) 캐릭터 동작 |
|---|---|---|
| **정지(Idle)** | 시간 표시 + `-`/`+` 버튼 + `▶`(시작) | 단일 정지 이미지 (1프레임) |
| **진행(Running)** | 카운트다운 시간 + `||`(일시정지) + `ㅁ`(중단/리셋) | 런닝머신 뛰기 루프 (4프레임, 1fps) |
| **일시정지(Paused)** | 멈춘 시간 + `▶`(재생) + `ㅁ`(중단/리셋) | `pause` 전용 동작 (없으면 stop 프레임 폴백) |
| **완료(Complete)** | 알림 발송됨 표시 + (탭 시 정지 상태로 복귀) | 승리 포즈 (2~3프레임) |

### 3-2. 타이머 조작 흐름
- **위젯에서 직접 조작**:
  - 정지 상태: `+`/`-` 버튼으로 시간 증감 (기본 단위 1분), `▶`로 시작
  - 진행 상태: `||` 일시정지, `ㅁ` 중단/리셋
  - 일시정지 상태: `▶` 재생, `ㅁ` 중단/리셋
- **버튼 동작 연결**: 각 버튼은 `PendingIntent`(`getBroadcast`)로 커스텀 액션을 발송하고, `AppWidgetProvider.onReceive`에서 액션을 받아 상태 전환·갱신 처리
- **앱 화면에서**:
  - 증감 단위를 자유롭게 설정 가능 (숫자패드 직접 입력)
  - 현재 진행 중인 타이머 상태가 앱 화면에도 동기화되어 표시됨
  - **타이머 위/아래 배치 전환 토글**(top/bottom) — 변경 시 위젯 즉시 갱신 (2-1 참조)
- **위젯 1개 = 타이머 1개** (다중 운영 미지원, 단순 구조)

### 3-3. 타이머 완료 시 동작
- 시스템 알림(Notification) 발송
- 캐릭터가 승리 포즈로 전환되어 위젯에 표시
- **완료 후 흐름**: 사용자가 위젯을 탭하면 정지(Idle) 상태로 복귀하며, **직전에 설정했던 시간 값을 유지**

## 4. 애니메이션 / 타이머 구동 구조 (확정 — 동작 안정성 핵심)

### 4-1. 프레임 갱신 방식
- **포그라운드 서비스 내부에서 Handler(`postDelayed`) 또는 코루틴으로 1초 틱**을 돌리고, 매 틱마다 (a) 남은 시간 계산 (b) 다음 프레임 PNG로 위젯 `ImageView` 교체를 수행한다.
  - ⚠️ `AlarmManager.setExactAndAllowWhileIdle`를 1초마다 재설정하는 방식은 시스템 throttle 때문에 1fps가 보장되지 않으므로 **프레임 갱신 용도로 사용하지 않는다.**
- **AlarmManager(inexact)**: 서비스가 비정상 종료되거나 기기 재부팅된 경우의 **복구 트리거 용도로만** 사용 (`SCHEDULE_EXACT_ALARM` 권한 불필요).
- 시간 계산은 `SystemClock.elapsedRealtime()` 기준 절대 시각(목표 종료 시각)을 저장해두고 매 틱마다 차이를 계산 → 절전 모드 진입/이탈 시에도 누적 오차 없음.

### 4-2. 화면 꺼짐 시 정책
- `Intent.ACTION_SCREEN_ON` / `ACTION_SCREEN_OFF`를 동적 BroadcastReceiver로 감지.
- **화면이 켜져 있을 때만** 1초 틱으로 프레임 교체(애니메이션) 수행.
- **화면이 꺼지면** 프레임 갱신은 멈추고, 남은 시간만 백그라운드에서 계속 추적(목표 종료 시각 기반). 화면이 다시 켜지면 현재 남은 시간에 맞춰 위젯 갱신.
- → 배터리 소모와 시스템 throttle 문제를 동시에 완화.

### 4-3. 권장 컴포넌트 구성
- `AppWidgetProvider`: 위젯 UI 렌더링 + 버튼 액션 수신(`onReceive`)
- `Foreground Service`: 타이머 진행/일시정지 중 시간 추적 + 1초 틱 프레임 갱신 (상시 알림 노출)
- `BroadcastReceiver`(동적 등록): 화면 ON/OFF 감지
- `AlarmManager`(inexact): 서비스 사망/재부팅 복구용
- `SharedPreferences`: 타이머 상태(목표 종료 시각, 설정 시간, 현재 상태값, 선택된 스킨ID, 배치 모드 layoutMode) 영속화 → 앱↔위젯 동기화

## 5. 권한 처리 (확정)

- **`POST_NOTIFICATIONS`** (Android 13+): 완료 알림이 핵심 기능이므로 앱 첫 실행 또는 첫 타이머 시작 시 런타임 권한 요청 흐름 포함.
- **포그라운드 서비스**: `FOREGROUND_SERVICE` 및 (targetSdk 34+) 적절한 `FOREGROUND_SERVICE_*` 타입 권한 선언, 진행 중 상시 알림 노출.
- **배터리 최적화 예외**: 일부 제조사(삼성/샤오미 등)에서 서비스 강제 종료 방지를 위해 배터리 최적화 예외 요청 안내 UX 포함(선택적 유도).
- **`SCHEDULE_EXACT_ALARM` 불필요**: 프레임 갱신을 핸들러 틱으로 처리하고 알람은 inexact 복구용으로만 쓰므로 정확 알람 권한은 선언하지 않는다.

## 6. 스킨 시스템 설계 (확장성 최우선)

### 6-1. 스킨의 구성 단위
하나의 스킨 = **캐릭터 + 전용 배경(런닝머신 디자인)이 묶인 세트**
(예: "감자" 스킨 → 감자 캐릭터 + 흙밭 배경 / "우주" 스킨 → 우주인 캐릭터 + 우주 배경)

### 6-2. 스킨 데이터 구조 (JSON 스키마)
```json
{
  "skinId": "potato",
  "name": "감자",
  "isFree": true,
  "character": {
    "stop": {
      "frames": ["idle_01.png"],
      "frameDurationMs": 1000
    },
    "running": {
      "default": {
        "frames": ["run_01.png", "run_02.png", "run_03.png", "run_04.png"],
        "frameDurationMs": 1000
      }
      // 향후 컨디션 분기 확장 시 추가 키 (예: "tired", "fresh") 삽입 예정
      // "tired": { "frames": [...], "condition": { "type": "remainingTime", "lte": 600 } }
    },
    "pause": {                       // 선택 — 일시정지 전용 동작. 생략 시 stop 프레임으로 폴백
      "frames": ["pause_01.png"],
      "frameDurationMs": 1000
    },
    "complete": {
      "frames": ["win_01.png", "win_02.png"],
      "frameDurationMs": 1000
    }
  }
}
```

**설계 포인트**:
- 캐릭터 상태 블록은 `states` → **`character`** 로, 정지 상태 키는 `idle` → **`stop`** 으로 명명한다.
- `character.running`을 단일 `frames` 배열이 아니라 **`default` 서브키를 갖는 객체**로 한 단계 감싸둔다.
  → 향후 "지친 모습/가뿐 모습" 같은 컨디션 분기를 추가할 때, `running` 객체 안에 `tired`, `fresh` 같은 키를 추가하고 분기 조건(`condition`)만 채워 넣으면 되도록 미리 구조를 열어둔다.
  → **1차 버전에서는 `default` 프레임셋만 사용하고 분기 로직 자체는 구현하지 않는다.**
- 일시정지 상태는 **선택적 `pause` 블록**으로 별도 동작을 줄 수 있고, 생략하면 `stop` 프레임으로 폴백한다.
- 상태별 프레임 수는 가변 배열로 처리 (stop=1장, running=4장, complete=2~3장 등 자유).
- 배경(런닝머신 디자인)은 스킨에 종속되므로 별도 분리하지 않고 캐릭터 프레임 이미지 자체에 합성하여 제작 (캐릭터+배경 합성된 PNG).

### 6-3. 에셋 폴더 구조
```
assets/skins/
  ├── potato/
  │   ├── skin.json
  │   ├── idle_01.png
  │   ├── run_01.png ~ run_04.png
  │   └── win_01.png ~ win_02.png
  ├── space/
  │   ├── skin.json
  │   ├── idle_01.png
  │   ├── run_01.png ~ run_04.png
  │   └── win_01.png ~ win_02.png
```
- 신규 스킨 추가 시 코드 수정 없이 `skins/` 폴더에 세트 추가 + `skin.json` 등록만으로 확장 가능하게 설계.
- 무료/유료 구분은 `isFree` 필드로 관리.
- 이미지 권장 해상도: 2×2 셀 영역 기준으로 hdpi~xxxhdpi 대응 가능한 충분한 해상도로 제작(런타임에 위젯 크기에 맞춰 스케일).

### 6-4. 결제 대비 자리 마련 (1차 버전에 포함 — 결제 로직 자체는 미구현)
1차 버전에서는 모든 스킨이 무료지만, **나중에 결제를 끼워넣기만 하면 되도록 다음 "자리"를 미리 만들어 둔다.** (실제 Billing 연동은 1차 이후 단계)
- **스킨 선택 화면을 잠금/해제 상태 표현이 가능한 구조로 설계**: `isFree` 값을 읽어 "해제됨/잠김"을 표시할 수 있게(1차 버전에선 전부 무료라 모두 해제 상태로 보이지만, 잠금 표시·구매 버튼이 들어갈 UI 자리는 마련).
- **"이 스킨을 사용 가능한가?"를 판단하는 함수를 한 곳에 집중**: 1차 버전에서는 `isFree == true`만 검사. 향후 이 함수 한 곳에서 `isFree == true || 구매됨(skinId)`로 한 줄만 바꾸면 결제 연동이 완성되도록.
- **구매된 skinId 목록을 저장할 자리를 데이터 모델에 미리 비워둔다**(예: `SharedPreferences`에 `purchasedSkinIds` 키). 1차 버전에서는 항상 비어 있음.

### 6-5. 캐릭터/타이머 영역 독립 선택 (구매는 테마 단위) — v2.2 확장

기존에는 한 스킨(테마)이 캐릭터+타이머를 묶어 통째로 적용됐다. 이를 **"구매는 테마(세트) 단위로 유지하되, 적용은 캐릭터/타이머를 따로"** 고를 수 있게 확장한다.

**비즈니스 원칙**: 캐릭터는 구매욕이 높고 타이머 영역은 부속 성격이라 가치가 비대칭이다. 그래서 **판매는 세트(테마) 단위**로 묶어 객단가·디자인 완성도를 지키고, **사용자 개인화는 적용 단계의 혼합 선택**으로 제공한다. (예: 감자 캐릭터 + 우주 타이머)

**데이터 모델**
- 단일 `selectedSkinId` → `selectedCharacterSkinId` + `selectedTimerSkinId` 두 키로 분리.
- 구버전에서 올라오면 기존 `selected_skin_id` 값을 양쪽에 복사해 마이그레이션(`TimerPreferences.load`).
- `purchasedSkinIds`는 **그대로 테마 단위** — 한 테마를 구매/보유하면 그 테마의 캐릭터·타이머 둘 다 사용 가능. 잠금 판정(`SkinAvailabilityChecker`)도 변화 없음.

**위젯 렌더링(`WidgetUpdater`)**
- 캐릭터 영역은 `selectedCharacterSkinId` 스킨의 `character`에서, 타이머 영역은 `selectedTimerSkinId` 스킨의 `timer`에서 가져온다.
- 각 영역의 비트맵은 그 영역 스킨의 `skinId` 폴더에서 로드되므로, 타이머 관련 렌더 함수에는 타이머 스킨을, 캐릭터 함수에는 캐릭터 스킨을 넘긴다(서로 다른 테마여도 충돌 없음).

**화면 구성** — 미리보기는 **전부 새 화면**(팝업 아님), 적용/상점 일관.
- **스킨 설정(적용) — `SkinSelectActivity`**: `[캐릭터][타이머]` 두 탭. 각 탭은 '사용 가능한(보유)' 스킨만 나열하고 영역별로 독립 선택. 선택 상태도 탭별로 따로 표시. 타이머 탭 썸네일은 `preview/{id}/thumb.png` 원격 로드(캐릭터 탭은 로컬 프레임). 항목 클릭 → 적용 미리보기 화면.
- **적용 미리보기 화면(신규)**: 보유분이므로 로컬 에셋으로 **상태 탭** 표시.
  - 캐릭터: 정지/진행/일시정지/완료 — 로컬 프레임 애니메이션.
  - 타이머: 정지/중단/진행/완료 — **로컬 타이머 스킨을 라이브 렌더**(보유=다운로드 완료라 박스/버튼/숫자 에셋이 로컬에 있음. 정적 미리보기 PNG는 없음).
- **상점(구매) — `SkinStoreActivity`**: 로컬+카탈로그 전체 테마 목록. 구매/미구매 포함 여부는 체크 토글(`showAll`). 테마가 캐릭터만/캐릭터+타이머인지 **태그로 식별**(로컬=`hasCustomTimer`, 원격=catalog `hasTimer`). **구매/다운로드 버튼은 목록 카드 + 상세 둘 다** 노출(노출 극대화).
- **상점 미리보기/상세 화면**: 미구매분이라 로컬 에셋이 없어 **CDN `preview/{id}/prevNN.png`를 스와이프 갤러리**로 보여준다(구별 없는 단순 가변 목록). 개수는 `prev01`부터 연속 **탐침(최대 prev30)**, 첫 결번에서 중단. 하단 [구매하기]/[다운로드] 버튼. (정지/진행중 고정 2장 토글은 폐기.)

**디자인 레포(daintyz_timer_characterList) 변경**
- 테마별 표시 에셋을 `preview/{id}/` 한 폴더에 모음: 썸네일 `thumb.png`(고정 파일명) + 미리보기 `prev01.png, prev02.png, …`(가변 개수, **prev01부터 빈 번호 없이 연속**). 캐릭터별 썸네일은 두지 않음(앱이 로컬 캐릭터 프레임 사용).
- prevNN은 **상점 갤러리 전용**(구매 전 미리보기). 적용 미리보기는 앱이 로컬에서 렌더하므로 prevNN과 무관 — 상태 의미 매핑 강제 없음(자유 컷).
- catalog.json에 **`hasTimer`** 필드 추가(없으면 true로 간주).
- **구매가 테마 단위이므로** `type` 필드/`timer_zip/`/character-optional 파서 같은 별도-상품용 구조는 **불필요**. 테마 zip은 지금처럼 캐릭터+타이머를 함께 담는다.

**결제**: 본 확장에서도 실제 Google Play Billing은 미구현(상세화면 구매 버튼은 스텁). 결제 성공 시 `purchasedSkinIds`에 테마 id 추가 → 다운로드 흐름만 이어붙이면 완성되도록 자리를 둔다.

## 7. 1차 버전 개발 범위

**포함**:
- 위젯 정지/진행/일시정지/완료 4상태 UI + 1fps 프레임 애니메이션
- 위젯 레이아웃 2×3 (타이머 2×1 + 캐릭터 2×2) 고정 크기, **타이머 위/아래 배치 전환 설정 포함**
- 감자 캐릭터 1종 (기본 무료 스킨)
- 타이머 조작: 위젯 `+`/`-`(기본 1분 단위)·`▶`·`||`·`ㅁ`
- 앱 화면: 증감 단위 자유 설정(숫자패드), 현재 타이머 상태 동기화 표시
- 완료 시 알림 + 승리 포즈, 탭하면 정지 상태(직전 설정시간 유지)로 복귀
- 포그라운드 서비스 기반 핸들러 틱 + 화면 ON/OFF 정책 + 재부팅/사망 복구
- 알림 런타임 권한 요청 흐름
- 스킨 JSON/폴더 구조는 확장성 있게 설계 (컨디션 분기 등 향후 기능을 염두에 둔 스키마)
- **결제 대비 자리 마련**(6-4): 잠금/해제 가능한 스킨 선택 UI, 사용 가능 판단 함수 일원화, 구매 목록 저장 자리 (결제 로직 자체는 미구현)

**제외 (1차 이후 확장)**:
- 결제 로직 구현(Google Play Billing 연동, 구매/복원/검증, 잠금 해제 동작)
- 컨디션 분기(지친/가뿐 모습 등 조건부 프레임 전환)
- 추가 캐릭터 유료 스킨 판매
- 위젯 다중 운영(타이머 여러 개 동시 진행)
- 위젯 리사이즈 대응

## 8. 구현 시 유의사항
- 위젯 갱신은 시스템이 보장하지 않으므로, 앱 재실행/위젯 재배치/재부팅 시에도 `SharedPreferences`에 저장된 목표 종료 시각·상태값으로 정확히 복구되도록 처리한다.
- 일시정지 시점에는 "남은 시간"을 저장하고, 재생 시 그 시점 기준으로 목표 종료 시각을 다시 계산한다.
- 프레임 인덱스도 상태 저장에 포함하거나, 화면 켜짐 시 시간 기준으로 프레임을 재계산하여 애니메이션이 자연스럽게 이어지게 한다.
